# audit/

The instrumentation layer for the analogy skill. Maps to §16 of the paper: *Measuring Impact*.

**Honest claim:** this scaffold makes the n=1 question askable. It does not, on its own, answer it. The skill is currently a single-user reference implementation. The audit shape is built so that if usage grows beyond n=1, the data structure already supports it.

---

## What lives here

| File | What it holds | Who writes it |
|---|---|---|
| `decisions.jsonl` | Append-only log: one JSON record per analogy invocation | The skill, every time it fires |
| `locks.jsonl` | Append-only log: pack-locks the user has pinned | The skill, when the user calls `lock this` |
| `check.py` | Stdlib-only analyser that scores the log and rolls up a report | Run manually or scheduled |
| `REPORT.md` | The latest generated rollup — committed so it can be diffed across runs | Generated by `check.py --write` |

No external dependencies. Python 3.10+.

---

## Schema — `decisions.jsonl`

One JSON object per line. Required fields marked **(req)**.

| Field | Type | Notes |
|---|---|---|
| `schema_version` | string | `"1.0"` (legacy) or `"1.1"` (current). Absent → treated as `"1.0"` by `check.py`. Only v1.1 records contribute to the validation gate; v1.0 records still count toward cadence and pack distribution. |
| `ts` **(req)** | ISO-8601 string | When the invocation fired |
| `session_id` | string \| null | Claude Code session ID if available |
| `trigger_conditions` **(req)** | list of strings | Which of the five fire conditions matched: `options` / `tradeoff` / `non_trivial` / `redesign` / `risk` |
| `domain_used` **(req)** | string | One of: `cooking`, `construction`, `home`, `travel`, `sport`, or a custom pack |
| `domain_override_reason` | string \| null | If the skill switched away from the default pack, why (one short line) |
| `components` **(req)** | object | Boolean flags for all six output elements: `{summary, mapping_table, simulation_prompt, limit_statement, mislead_tag, second_opinion_pass}`. The last two are the §15A.1 defences against class 3 (false-confident) — see `SKILL.md`. A full v1.1 rendering ships all six. |
| `frame_rejection` **(req)** | boolean | Did the user reject the option set after the analogy reframe? §8's claim — the most distinctive measure |
| `user_feedback` | string \| null | One of the six user-callable verbs from `SKILL.md`: `flag_unfamiliar` \| `flag_misleading` \| `flag_bleed` \| `drop_analogy` \| `redo:<domain>` \| `redo_scale:<scale>` \| `lock` \| `null`. Maps to classes 1, 2, 4, 5, 6, 6, 7 respectively. Class 3 (false-confident) is post-hoc only — see `outcome_label` below. |
| `decision_class` | string \| null | `build` \| `triage` \| `audit` \| `explanation` — matches paper §8/§9/§10 case types |
| `outcome_label` | string \| null | Filled in retrospectively: `comfort` \| `quality` \| `both` \| `false_confident` \| `unknown`. `false_confident` is the class 3 label — the user simulated confidently against the analogy but the substrate behaved differently. Honest about post-hoc labelling: this field captures judgements made after the decision, not at invocation. |
| `notes` | string \| null | Free-form. Optional. |

The `components` object is what makes the audit work — `check.py` rolls up how often the skill ships all six elements vs how often it degrades. The gate threshold is 80% all-six. A consistent 4-of-6 pattern means the §15A.1 defences (`mislead_tag` + `second_opinion_pass`) are dropping out — the skill is running in v1.0 shape inside a v1.1 frontmatter, not v1.1 itself.

## Schema — `locks.jsonl`

| Field | Type | Notes |
|---|---|---|
| `ts` | ISO-8601 string | When locked |
| `pack` | string | Pack name |
| `problem_signature` | string | A short user-supplied or skill-inferred signature: e.g. `"cumulative-budget"`, `"multi-party-time-bound"`, `"team-coordination"`. The skill uses these to suggest a pack on similar future problems. |
| `notes` | string \| null | Why this lock |

## Running `check.py`

```
python3 audit/check.py                     # print rollup to stdout
python3 audit/check.py --write             # write rollup into REPORT.md
python3 audit/check.py --since 2026-05-01  # filter by ISO date
python3 audit/check.py --pack construction # filter by pack
```

The rollup answers four questions, in order of paper §16's claims:

1. **Decision speed proxy** — invocations per day, time-to-close median (if logged)
2. **Frame-rejection rate** — `frame_rejection: true` / total
3. **Component completeness** — % of invocations shipping all 4 of (summary / mapping / simulation / limit)
4. **Comfort-vs-quality split** — distribution of `outcome_label` across logged invocations

Each of these is a **proxy**, not the construct itself. §16 calls this honestly; so does this report.

---

## What the audit cannot tell you

- **Whether the analogy was structurally sound.** That's a §12 question and requires reading the mapping. No statistical signal in the log will catch a wrong-but-coherent mapping (false-confidence failure).
- **Whether the decision was actually better than the no-layer counterfactual.** §16's control conditions require comparison data the log doesn't have. Outcome labels are post-hoc and self-reported.
- **Whether engagement improvement (comfort) is masquerading as quality improvement.** This is the paper's central open question. The log is shaped to surface the question; only deliberate review answers it.

---

## How to use this if you're not the author

The skill works without the audit running. Audit is for when you want to ask "did the analogy layer actually help?" and have data more substantive than memory. If you're a single user trying the skill for a week, log a few invocations by hand into `decisions.jsonl`, then run `check.py`. If you're integrating the skill into an AI tool, wire the log write into the same code path that produces the four-component output.
