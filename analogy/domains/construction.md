# construction — domain pack

> Construction holds layered roles with persistent obligations over time. Different parties carry different parts of risk and reward; the contract structure determines who answers when something fails three months after handover. This is the pack to reach for when the problem is *who carries which risk, who pays for what, what triggers what, and what happens over time* — the §9b case from the paper.

---

## Roles (who does what)

| Concept | Construction term | Use when |
|---|---|---|
| User | Developer / client | Pays for the build, signs off, holds the land |
| Expert (any tier) | Specialist trade / contractor | Owns one trade (electrical, structural, mechanical); doesn't reach across |
| Meta-expert | Architect / lead designer | Designs the system; doesn't build it |
| Adversarial reviewer | Building inspector / independent surveyor | Catches what the trades and the architect both missed |
| Subagent | Apprentice / labourer | Does the prep and the simple runs; doesn't make calls |
| Domain expert | Specialist consultant (acoustic / fire / planning) | One narrow technical question, deep expertise |
| Sub-expert pair | Lead trade + second opinion | Two voices on the same trade, deliberately differing on one axis |

## Gear (the tools and objects)

| Concept | Construction term | Use when |
|---|---|---|
| Persona file | Scope of works / trade contract | What this trade is hired to do, written down |
| Calibration card | Spec sheet / building regs reference | The reference standard the work must meet |
| Instinct file | The trade's toolbox + muscle memory | What fires automatically before conscious thought |
| Notepad / reflex bank | Site notebook / day book | Reminders the trade scribbles between rooms |
| Working memory (post-/dream) | Lessons learned from the last project | What this trade brings forward from previous builds |
| Hook | Inspection trigger / cert renewal | Fires automatically when stage X is reached |
| MCP server | Builders' merchant / supply yard | Where ingredients arrive from; not on-site by default |
| Knowledge pack | Method statement / trade standards (NHBC, NEC) | Written procedure for one class of work |
| Skill | A trade qualification (wiring, plastering) | Portable across sites |
| Brief | Architect's brief / planning consent | What this build is, before the first sod is cut |
| Project | The build | Lives or dies as a unit; certified at the end |
| Decision queue | Snag list / RFI log | Open questions waiting on the right party |

## Rhythms (how time and pace work)

| Concept | Construction term | Use when |
|---|---|---|
| AWE (token effort) | Labour hours + materials | The actual cost of doing the thing |
| Session | A day on site / a work package | One bounded run of activity |
| Session-open | Toolbox talk / pre-start briefing | Last reads before the trades pick up tools |
| Session-close | End-of-day site clean + handover | Tidy, sign off the day's work, note what's outstanding |
| Calibration cycle | Stage inspection / milestone review | Walk the site at first-fix / second-fix / practical completion |
| Coaching pass | Mentoring with a journeyman | Targeted feedback on one trade, not the toolbox talk |
| Drift | Scope creep / spec deviation | What's getting built isn't what's on the drawings |
| Cross-triage | Independent surveyor visit | Outside eyes on the trades' work |

## Verbs (what people do)

| Concept | Construction term | Use when |
|---|---|---|
| Recommend | Submit drawing / RFI | Put the proposal in front of the principal |
| Veto | Reject / require rework | "This isn't to spec — pull it out" |
| Defer | Hold pending inspection / variation order | Don't proceed until the question is answered |
| Approve | Sign off / issue certificate | The work passes, the stage completes |
| Calibrate | Re-measure / re-spec | Adjust the drawings to what's actually buildable |
| Iterate | Rework / re-fit | Strip out and redo |
| Build (new expert) | Bring on a new trade | Award the contract, hand over the scope |
| Retire (an expert) | Decommission / strip out | Off the site; that trade's work is gone |
| Adversarial pass | Independent structural review | Bring in someone who didn't design or build it |
| Stance protocol | Compliance / value engineering / future-maintainability | Three different lenses every drawing is checked under |

## Freeform mappings

- **The site** — the chat window or the codebase. Everything happens here; outside the hoarding is somebody else's problem.
- **Foundations** — the schema, the base architecture. Get this wrong and everything above moves.
- **First fix** — the rough-in. Wiring, plumbing, structural systems. Done before the walls close.
- **Second fix** — finishes. Switches, taps, paintwork. Cosmetic until you live in it.
- **Snagging** — the drift-detection pass at the end. Catalogue of what's almost right.
- **Variation order** — a scope change with a paper trail. Both sides sign.
- **RFI** (Request for Information) — the open question waiting on the architect.
- **The drawings** — the spec. The single source of truth for what is supposed to exist.
- **Practical completion** — the ship date. Usable, not yet finished.
- **Defects liability period** — post-launch warranty window. Things will still go wrong; the contract says who fixes them.
- **As-built** — what actually exists vs what was drawn. The honest record after drift.
- **The handover pack** — documentation passed to the operator at completion. Drawings, certs, manuals.

## When this analogy doesn't fit

- **Short-cycle iterative work.** Cooking is faster and more responsive. Construction implies cure times.
- **Pure run-time / live-service problems.** Once a building is up, the analogy gets thin — operations are a different metaphor.
- **Throwaway POCs.** You don't pour foundations for a tent. Reach for cooking or home.
- **Brand / aesthetic / voice problems.** Construction is structural; it doesn't carry feel and finish well.
